## Batch 1 — API Key settings modal + header button + backend per-request key override

**Background**: Users need a settings button in the top-right corner of the app to enter their own API keys. The system should use those keys in preference to the `.env` defaults; if a field is left blank, the server default is used transparently.

---

### 1) `backend/app/core/request_context.py` (new file)

- Change: Added four `ContextVar`s:
  - `openai_key_override`
  - `google_maps_key_override`
  - `gemini_key_override`
  - `tavily_key_override`
- Effect: Provides per-request API key override capability within Python async context.

---

### 2) `backend/app/main.py`

- Location: Middleware section (after approx. line 20)
- Change: Added `@app.middleware('http')` handler `extract_api_keys`:
  - Reads `X-OpenAI-Key` / `X-Google-Maps-Key` / `X-Gemini-Key` / `X-Tavily-Key` from request headers
  - Sets corresponding `ContextVar`s for the current request context
- Effect: All downstream services can read the user-provided key for that request.

---

### 3) `backend/app/core/config.py`

- Location: After `get_settings()` (approx. line 27)
- Change: Added four helper functions:
  - `get_effective_openai_key()`
  - `get_effective_google_maps_key()`
  - `get_effective_gemini_key()`
  - `get_effective_tavily_key()`
- Logic: Returns `ContextVar` override value if set, otherwise falls back to `.env` configured value.
- Effect: Unified "override-first, default-fallback" key resolution; callers don't need to know the source.

---

### 4) `backend/app/agents/triage_agent.py`

- Location: After `TriageAgent.__init__` (approx. line 77)

  - `_is_enabled()`: Returns `True` if any OpenAI key is available (override or default).

- Location: `_invoke_gpt_routing` (approx. line 170)

- Location: `assess()` (approx. line 404)
- Change: Replaced `if not self.enabled or self.client is None:` with `if not self._is_enabled():`.

- Effect: Each request uses the corresponding OpenAI key; keys provided by the frontend take effect immediately without restarting the server.

## Batch 4 — Appointment time slot picker in booking page

**Background**: The booking page previously showed only a single suggested visit time with no user selection. User requested a simulated hospital database time slot picker where some slots are shown as fully booked and unavailable.

### 1) `frontend/src/App.jsx`

- Location: `BookingView` component (approx. line 1018) — full rewrite
- Changes:
  1. Added `generateTimeSlots(providerName)` function above the component:
     - Generates 11 slots per day across 3 days (tomorrow through day after tomorrow)
     - Slot times: 9:00 AM – 11:00 AM and 2:00 PM – 4:30 PM in 30-minute increments
     - ~30% of slots marked as "booked" using a deterministic hash of the provider name, so the same provider always shows the same availability pattern (simulating a real DB)
  2. Added time slot grid UI inside `BookingView`:
     - Three day sections with labeled date headers
     - 4-column grid of slot buttons per day
     - Available slots: blue, clickable, highlights on selection with a scale-up effect
     - Booked slots: grayed out, strikethrough, "Full" label, `disabled` — cannot be clicked
     - Selected slot shows a confirmation banner below the grid
  3. Submit button is disabled (gray, "Please select a time slot") until both a time slot and patient contact info are filled in; enabled state shows "Confirm Booking Intent".
  4. `onSubmit` call now passes `selectedSlot?.iso` as the second argument.

- Location: `handleBookingSubmit` (approx. line 1548)
- Changes:
  1. Added `preferredTime` as second parameter.
  2. `resolvedTime` uses `preferredTime` first, then falls back to `provider.next_available_slot`, then a 2-hour offset from now.
  3. `preferred_time` in both the API call and the `localStorage` record now uses `resolvedTime`.
- Effect: The appointment time selected by the user is correctly persisted in the booking record and shown in the "My Appointments" panel on the home page.
