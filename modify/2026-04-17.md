  3. `preferred_time` in both the API call and the `localStorage` record now uses `resolvedTime`.
- Effect: The appointment time selected by the user is correctly persisted in the booking record and shown in the "My Appointments" panel on the home page.

## Batch 5 — Simulated hospital slot database (backend-driven time slot picker)

**Background**: The previous time slot picker was entirely hardcoded in the frontend using a hash of the provider name. User requested a proper backend slot database that the frontend queries, so slots are read from a real data source rather than computed locally.

### 1) `backend/app/services/slot_service.py` (new file)

- Change: New service that simulates a hospital appointment database.
  - `initialize_slots(provider_id)`: generates 11 × 3-day slot pool (9:00–11:00 AM and 2:00–4:30 PM, 30-min increments) for a provider. ~30% of slots are pre-marked as `booked` using a deterministic seed derived from `provider_id`, so each provider always has a stable, unique availability pattern.
  - `get_or_init_slots(provider_id, store_ref)`: returns existing slots from the store or initializes them on first access.
  - `mark_slot_booked(provider_id, slot_id, store_ref)`: atomically marks a slot as `booked`; returns `False` if the slot was already taken (double-booking prevention).
- Each slot object: `slot_id`, `iso`, `date_label`, `time_label`, `status` (`available` | `booked`).

### 2) `backend/app/services/store.py`

- Change: Added `self.provider_slots: dict[str, list[dict]]` to `InMemoryStore` — the in-memory slot database, keyed by `provider_id`.

### 3) `backend/app/api/v1/booking.py`

- Change: Added `GET /api/v1/booking/slots?provider_id=xxx` endpoint:
  - Calls `get_or_init_slots()` and returns the full slot list for the provider.
  - Frontend uses this to populate the time slot picker.
- Change: Updated `POST /api/v1/booking/intents` to:
  - Accept optional `slot_id` field.
  - Call `mark_slot_booked()` when `slot_id` is provided.
  - Return `409` if the slot was already booked (race condition / double-booking).

### 4) `backend/app/schemas/booking.py`

- Change: Added `slot_id: str | None = None` field to `BookingIntentRequest`.

### 5) `frontend/src/api.js`

- Change: Added `getProviderSlots(providerId)` → `GET /api/v1/booking/slots?provider_id=...`.

### 6) `frontend/src/App.jsx`

- Change: Imported `getProviderSlots` from `./api`.
- Change: Replaced `generateTimeSlots()` frontend function with backend-driven slot loading in `BookingView`:
  - `useEffect` calls `getProviderSlots(provider.provider_id)` on mount.
  - Shows a loading spinner ("Fetching available slots…") while the request is in flight.
  - Shows an error message if the request fails.
  - `groupSlotsByDay()` helper groups the flat slot list returned by the API into per-day sections for display.
  - Slot UI is unchanged (blue = available, gray + "Full" = booked, highlighted = selected).
- Change: `onSubmit` now passes three arguments: `patientContact`, `selectedSlot.iso`, `selectedSlot.slot_id`.
- Change: `handleBookingSubmit` updated to accept `slotId` and pass it as `slot_id` in the booking intent payload.

## Batch 6 — Confidence boost scoring + force-recommend button in follow-up stage

**Background**: The follow-up symptom stage was too long, with confidence sometimes never rising. Two improvements: (1) each suggested symptom now carries an AI-assigned `confidence_boost` value (1–4) that immediately raises the displayed confidence when the patient selects it; (2) a "I've described all my symptoms — get recommendation now" button lets the patient skip the confidence gate entirely.

### 1) `backend/app/agents/triage_agent.py`

- Location: `SYSTEM_PROMPT` — `likely_symptoms` description
- Change: Changed `likely_symptoms` format from a list of plain strings to a list of objects:
  `{"label": "<first-person statement>", "confidence_boost": <1–4>}`.
  Higher `confidence_boost` = selecting this symptom would most help narrow down the correct department.
- Location: `assess()` (GPT path) — `likely_symptoms` normalisation
- Change: Parses `likely_symptoms` items as objects; extracts `label` and `confidence_boost` (clamped to 1–4); falls back to `confidence_boost=1` for plain strings or fallback data. Returns `likely_symptoms` as `list[{label, confidence_boost}]` in both GPT and fallback paths.
- Location: `generate_follow_up_questions()`
- Change: Pool is now built from `{label, confidence_boost}` objects. Each generated question dict includes `confidence_boost` field.

### 2) `backend/app/schemas/triage.py`

- Change: Added `confidence_boost: int = 1` to `FollowUpQuestion`.
- Change: Added `force_recommend: bool = False` to `SubmitAnswersRequest`.

### 3) `backend/app/api/v1/triage.py`

- Location: `submit_session_answers()` — before the normal answer submission block
- Change: Added "Force recommend" branch: when `payload.force_recommend` is `True`, saves the submitted answers, calls `triage_agent.assess()` immediately, sets session to `TRIAGE_READY`, and returns without any confidence gate check.

### 4) `frontend/src/App.jsx`

- Location: Question normalisation (session creation + follow-up submit response)
- Change: Added `confidenceBoost: q.confidence_boost || 1` when mapping API questions to local state.
- Location: `FollowUpView` component
- Change: Computes `localBoostPercent` = sum of `confidenceBoost` for all currently selected symptoms. Displayed confidence bar and percentage show `serverConfidence + localBoostPercent` in real time as the user selects symptoms. Each selected symptom with `confidenceBoost > 1` shows a small `+N%` badge.
- Change: Added second button below the main submit button: "I've described all my symptoms — get recommendation now". Calls `onSubmit({ ..., forceRecommend: true })`.
- Location: `handleFollowUpSubmit()`
- Change: Added `forceRecommend` parameter; passes `force_recommend: forceRecommend` in the `submitAnswers` payload.

## Batch 7 — Fix confidence regression after submitting selected symptoms

**Background**: After selecting a +4% symptom, the displayed confidence sometimes dropped (e.g. 80% → 77%) because the server re-ran GPT independently and returned a lower value, overwriting the locally boosted display.

### 1) `backend/app/api/v1/triage.py`

- Location: `submit_session_answers()` — normal submit + force-recommend branches
- Change: Added `_apply_boost()` inline helper that:
  1. Looks up `confidence_boost` for each selected (`value=True`) answer in `session['questions']`
  2. Uses `max(prev_confidence_percent, gpt_confidence_percent)` as a floor — confidence never regresses from a prior round
  3. Adds the selected boost sum on top of that floor (capped at 100)
- Effect: Confidence now always increases (or holds) when the patient selects symptoms; GPT variance can no longer cause a regression.
