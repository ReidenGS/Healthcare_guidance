## 第一批：API Key 设置弹窗 + 前端 Header 按钮 + 后端 Per-Request Key Override

**背景**：用户希望在前端右上角放置一个设置按钮，点击后弹出小窗口让用户填写自己的 API Key，系统优先使用用户填写的 Key，未填写则使用 `.env` 中的默认值。

---

### 1) `backend/app/core/request_context.py`（新增）

- 位置：整体新增
- 变更：新增四个 `ContextVar`：
  - `openai_key_override`
  - `google_maps_key_override`
  - `gemini_key_override`
  - `tavily_key_override`
- 功能效果：提供 Python 异步上下文中的每请求 Key 覆盖能力。

---

### 2) `backend/app/main.py`

- 位置：中间件区（约第 20 行后）
- 变更：新增 `@app.middleware('http')` 装饰器 `extract_api_keys`：
  - 读取请求头 `X-OpenAI-Key` / `X-Google-Maps-Key` / `X-Gemini-Key` / `X-Tavily-Key`
  - 通过 `ContextVar.set()` 注入当前请求上下文
- 功能效果：后续所有服务在同一请求中均可读取到前端传入的 Key。

---

### 3) `backend/app/core/config.py`

- 位置：`get_settings()` 之后（约第 27 行后）
- 变更：新增四个 helper 函数：
  - `get_effective_openai_key()`
  - `get_effective_google_maps_key()`
  - `get_effective_gemini_key()`
  - `get_effective_tavily_key()`
- 逻辑：优先返回 `ContextVar` 覆盖值，无覆盖时回退到 `.env` 配置值。
- 功能效果：统一"有覆盖用覆盖，无覆盖用默认"的取值逻辑，调用方无需关心来源。

- 功能效果：每次请求时使用对应的 OpenAI Key，前端传入 Key 时即时生效，无需重启服务。
