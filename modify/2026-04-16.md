## Batch 3 — Fix RAG online search falsely escalating simple symptoms

**Background**: After introducing RAG, entering "cough" routed directly to Urgent Care, bypassing the follow-up questioning flow.

**Root cause**:
- Offline vector store had no match for "cough" → triggered `_search_redflag_online("cough")`
- Tavily returned ER-scenario content for cough
- That content was injected into GPT as "Red-flag retrieval context"
- GPT was influenced toward aggressive routing with inflated confidence, skipping follow-up

### 15) `backend/app/agents/triage_agent.py`

- Location: `assess()` Step 1–2 (approx. line 387)
- Change: Replaced `search_with_online_fallback()` with a three-tier independent logic:

  **Tier A** (offline RAG hit): Inject confirmed red-flag evidence, skip all network searches.

  **Tier B** (no offline hit, but symptom description ≥ 4 words or severity ≥ 5): Run department routing search first, then overlay online red-flag search as **supplementary** context with prompt annotation "do NOT override severity rules".

  **Tier C** (simple/vague input like "cough"): Use **only** department routing search; no online red-flag search; preserves the low-confidence → follow-up flow.

- Removed redundant `severity` local assignment in Step 4 (already defined at function top).
- Effect: Simple symptoms are no longer aggressively routed by ER-focused search content; detailed or serious descriptions still benefit from online red-flag assistance.

- Removed the now-unused `search_symptom_reference` import.
- Added `INITIAL_CONFIDENCE_CAP = 72`: when `answered_count == 0`, confidence is capped at 72, ensuring initial assessment always falls below the 80 threshold and requires at least one follow-up round.
- Effect: Simple/vague symptoms (e.g. "cough") yield initial confidence 40–55, reliably triggering follow-up.

## Batch 6 — Insurance demo logic refactor

**Background**: Project simulates access to insurer databases for demo purposes. Rewrote to use a deterministic group split (half always in-network, half always out-of-network) to showcase price comparison.

### 19) `backend/app/services/insurance_service.py`

**Core change**: Fully rewrote `estimate_insurance_from_mock_db()`, removing provider ID/name/keyword matching and AI reviewer call.

**Grouping rule** (based on insurer index in `insurance_networks.json`):

| Index | Insurer | Group |
|-------|---------|-------|
| 0 | Aetna PPO | In-Network |
| 1 | Blue Cross Blue Shield | Out-of-Network |
| 2 | UnitedHealthcare | In-Network |
| 3 | Medicare / Medicaid | Out-of-Network |

**In-network group (`idx % 2 == 0`)**: returns in-network cost, out-of-network cost for comparison, in-network breakdown, coverage ratio, and savings notice.

**Out-of-network group (`idx % 2 == 1`)**: returns out-of-network cost with no discount, OON breakdown, `coverage_ratio: 'No coverage'`, and notice recommending a different insurer.

Removed: `_deterministic_support()`, `_ai_support_decision()`, `_build_ai_client()`, related LangChain/langchain_openai imports.

Retained: `estimate_insurance()` (legacy simple estimator for backward compatibility), `_to_breakdown_lines()`, `_normalize()`, `_load_network_db()`.

## Batch 8 — Confidence fully determined by GPT; removed all formula calculations

**Background**: User requested that no hardcoded formulas be used for confidence scoring; GPT's `confidence_percent` should be trusted entirely.

### 23) `backend/app/agents/triage_agent.py`

  - `confidence_percent` controls whether follow-up continues (< 80 continues, ≥ 80 exits to recommendation).

- Removed `if answered_count == 0: calibrated_percent = min(...)` cap logic.
- Changed to `calibrated_percent = percent` (direct use of GPT value).
- Effect: AI autonomously determines confidence based on symptom specificity and follow-up answer quality.
