# 医疗导航项目业务逻辑（当前实现）

## 1. 业务目标
1. 根据患者描述判断是否能高置信度推荐具体就诊科室。
2. 在不确定时，通过 AI 引导补充症状而不是直接给模糊建议。
3. 在需要就医时，继续给出附近机构、保险费用、预约意向链路。
4. 在无需就医时，允许用户直接结束本轮，不进入医疗机构匹配。

## 2. 端到端流程
1. `PROFILE`：用户填写基础信息与位置信息（地址/城市/ZIP 任一可填）。
2. 位置解析：
   - 先走 `/geo/geocode`
   - geocode 失败则尝试浏览器 GPS
   - 若 GPS 也失败，提示用户开启定位或补充地址
3. `INTAKE`：提交主诉 + 严重度。
4. GPT 分诊首轮：
   - `confidence_percent > 80` -> `TRIAGE_READY`
   - 否则 -> `FOLLOW_UP`
5. `FOLLOW_UP`：
   - 每轮 4 个第一人称候选症状
   - 用户选择结果回注到下一轮 prompt
   - 已问症状去重，避免重复选项
6. `TRIAGE_READY` 推荐判定：
   - 常规：用户同意后进入机构搜索
   - 特殊：若 `Primary Care + visit_needed=false`，直接结束分支（回首页或重写症状）
7. 需要就医分支：机构匹配 -> 保险估算 -> 预约意向 -> 完成。

## 3. 关键判定规则

### 3.1 置信度门控
- 只有 `confidence_percent > 80` 允许进入推荐环节。

### 3.2 Follow-up 规则
- 每轮 4 条候选症状。
- 必须是患者视角陈述句。
- 使用临时文件记忆已问症状并去重。

### 3.3 是否就医判定（新增）
- 字段：`visit_needed`（来自推荐接口）。
- 业务解释：
  - `true`：建议继续就医流程。
  - `false`：当前更偏向自我护理，不进入 provider 阶段。

## 4. 关键数据对象
1. 推荐结果：
   - `department`
   - `care_path`
   - `confidence_percent`
   - `visit_needed`
   - `reasons`
2. 位置结果：
   - `normalized_address`
   - `lat/lng`
   - `source`
3. 履约结果：
   - `providers[]`
   - `insurance result`
   - `booking_intent_id`
   - `timeline`

## 5. 一致性约束
1. 前端不得绕过 `visit_needed=false` 分支去调 provider 接口。
2. 前端 follow-up 题目必须完全使用后端返回的 `questions[]`。
3. 后端不得假设前端一定有精确地址，需兼容 geocode + GPS 路径。
4. 所有对外文案必须声明“导航建议，不是医疗诊断”。

## 6. 业务收益
1. 提高“科室推荐”可解释性与可执行性。
2. 降低误导性过度分流（通过 follow-up 和 visit_needed 分支）。
3. 将分诊、机构、费用、预约串成闭环，支持真实落地。
